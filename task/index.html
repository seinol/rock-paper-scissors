WED1 - Testat

Ziele des Testates

In der Vorlesung behandelte Aspekte vertiefen und anwenden.

Allgemeine Hinweise

  • Aufgrund von der aktuellen Situation ist das Testat eine Einzelarbeit.

Aufgabenbeschreibung

Ihre Aufgabe ist es, das Spiel Schwere Stein Papier (Brunnen Streichholz) zu implementieren. Das Testat ist in zwei Teile aufgeteilt.

Teil 1: Schwere Stein Papier mit In-Memory

Ziel vom Teil 1 ist es alle Features zu implementieren und die Daten im Memory zu halten.

Folgende Features sind nötig um den ersten Teil vom Testat zu bestehen:

  • Startseite
    • Rangliste
    • Eingabe vom Namen (Required)
    • Spiel Starten
  • Spielseite
    • Spieler-Name darstellen
    • Darstellen der möglichen Hände
    • Auswahl der Hand
    • Anzeigen der Wahl der der PC getroffen hat.
    • Darstellen vom Resultat.
    • Zwischen den einzelnen Spielen ist eine Wartezeit einzuhalten.
    • History: Die History zeigt von der aktuellen Spiel-Session alle vergangen Spiele an.
    • Zurück zu Startseite.

Teil 2: Schwere Stein Papier mit Server-Anbindung

Ziel vom Teil 2 ist es dem Benutzer anzubieten ob er mit dem Server Kommunizieren möchte oder Lokal Spielen möchte.

Nachfolgend die API vom Server:

METHODE URL Beschreibung Rückgabe
GET /ranking Daten für die Rangliste Beispiel
GET /play?playerName=[name]
&playerHand=[playerHand]		 Lässt den Server ein Spiel auswerten Beispiel

Folgende Features sind nötig um den zweiten Teil vom Testat zu bestehen:

  • Startseite
    • Wechseln vom Lokalen und Server Modus
    • Die Webseite soll sich der gewählte Modus merken
    • Darstellen vom Ranking vom Server oder Lokal
  • Spielseite
    • Das Spiel soll mit dem korrekten Services nutzen ( Server oder In-Memory ).

Vorgehen

  1. Erstellen Sie ein neues HTML-Projekt.
  2. Versuchen Sie Feature um Feature zu implementieren. z.B. Das Ranking darstellen.
  3. Fragen Sie den Übungsbetreuer wie Sie weitervorgehen können.

Kriterien/Anforderungen

  • HTML5, CSS und JavaScript Standardkonformität muss erfüllt werden.
  • Überprüfen Sie ihre Arbeit mit dem W3C Validator vor der Abgabe und beheben gefundene Fehler.
  • Ihre Lösung muss in den neusten Versionen von 2 der 4 grossen Browsern (FF, GC, IE, AS) getestet worden sein und ohne Fehler funktionieren.
  • Die Lösung soll sinnvoll gestyled sein.
  • Der JavaScript-Code muss sinnvoll aufgeteilt sein.
  • Die Lösung soll eine "Single Page" sein.

Review & Abgabe

Senden Sie ihre Lösung per E-Mail an Ihren Betreuer mit folgendem Inhalt:

  • Ihre Lösung als ZIP-File
  • Angabe der getesteten Browser mit Versionsnummer
  • Screenshot der W3C Validation.

Nachkorrektur

Jede Person hat die Möglichkeit für eine Nachkorrektur. Diese erfolgt eine Woche später. Eine zweite Nachkorrektur wird nicht akzeptiert. Nachkorrekturen müssen entweder präsentiert werden, oder benötigen nur eine erneute Abgabe der korrigierten Lösung per Mail an den Betreuer. Der Betreuer entscheidet was nötig ist.
